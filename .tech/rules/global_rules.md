# Global Rules (전역 코딩 및 보안 규칙)

이 규칙은 `GPHarness` 파이프라인에서 작업을 수행하는 **모든 에이전트(Planner, Generator, Evaluator)**가 예외 없이 지켜야 하는 최우선 가이드라인입니다.

## 0. 언어 및 작성 규칙
- 문서화, 커밋 메시지 본문, 주석, 설계안은 **한국어(Korean)**로 통일합니다.
- 식별자(함수, 클래스, 변수명)는 **의미있는 영문(Meaningful English)**을 사용합니다.

## 1. 설계 및 구현 (Single Responsibility)
- **목적 기반 구현**: 모든 코드는 단일 책임 원칙(SRP)을 준수해야 합니다.
- **가정 기반 중간 솔루션**: 모호하거나 불확실한 요구사양이 있을 경우에 한하여 아래 포맷으로 대응 후 확인을 요청합니다.
  - `- Assumption:` 내가 설정한 가정
  - `- Risk:` 이 가정이 틀렸을 때의 잠재적 문제
  - `- Fallback:` 틀렸음이 확인될 시 적용할 대체 구조

## 2. 보안 가이드라인 (Security First)
- **민감 정보 하드코딩 금지**: API Key, 패스워드는 코드나 PRD 등에 직접 명시하지 않으며 환경 변수나 Secret Manager를 사용합니다.
- **외부 입력 검증 및 정제**: XSS, SQL Injection 방지를 위해 모든 데이터 입력/출력 단에서 Validation 및 Sanitization을 적용합니다.

## 3. 안정성 및 에러 핸들링 (Observability & Error Handling)
- **에러 핸들링**: 
  - 예외 상황을 조용히 무시(Swallow)하지 않습니다. 에러 발생 시 로그를 명시적으로 남깁니다.
  - 외부 통신에는 반드시 타임아웃을 설정합니다.
- **관측 가능성(Observability)**: 주요 처리 로직의 시작/끝, 분기 처리 지점, 에러 시에는 로그, 메트릭, 트레이싱 중 1개 이상의 형태로 추적 가능성을 부여합니다.

## 4. 커밋 및 형상 관리 규칙
- **가벼운 커밋(Small Unit)**: 논리적으로 독자적인 단위는 독립적인 커밋으로 분리합니다.
- **Conventional Commits**:
  - `feat:`, `fix:`, `refactor:`, `docs:`, `test:`, `chore:`

> **[우선순위 지침]**
> 규칙 충돌 시 다음의 우선순위를 따릅니다.
> 1) 보안 / 데이터 유실 방지
> 2) 안정성 / 롤백 가능성
> 3) 요구사항 정확성
> 4) 성능
> 5) 문서화 / 포맷
